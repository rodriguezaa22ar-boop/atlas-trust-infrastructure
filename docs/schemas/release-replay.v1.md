# Schema Contract: atlas.release_replay.v1

## Surface

```bash
atlas release replay <packet> --json
```

## Purpose

`atlas.release_replay.v1` records the result of replaying a retained release
packet from an isolated checkout of the commit recorded inside the packet.

The replay result is metadata-only. It records the packet path, commit, replay
checkout identity, check statuses, cleanup state, metadata boundary, known
limitations, and non-guarantees. It must not embed QA logs, runtime artifacts,
target evidence, secrets, customer data, packet captures, or raw business
records.

## Required Fields

- `schema_version`: must be `atlas.release_replay.v1`.
- `generated`: UTC timestamp generated by Atlas.
- `metadata_only`: must be `true`.
- `raw_artifacts_embedded`: must be `false`.
- `overall`: `verified` when all required replay checks pass.
- `packet.path`: release packet path supplied to replay.
- `packet.commit`: commit recorded in the release packet.
- `replay_checkout.branch`: temporary replay branch name.
- `replay_checkout.path`: isolated checkout path used during replay.
- `replay_checkout.keep_worktree`: boolean.
- `replay_checkout.cleanup`: `removed` or `kept`.
- `checks.checkout.status`: `ok` after the isolated checkout reaches the
  packet commit.
- `checks.qa.status`: `ok` when QA ran successfully, or `skipped` when
  `--skip-qa` was used.
- `checks.v1_status.status`: `ok` after `atlas v1 status --strict` passes in
  the replay checkout.
- `checks.release_verify.status`: `ok` after `atlas release verify <packet>
  --commit <commit>` passes from the replay checkout.
- `metadata_boundary.stores`: metadata classes recorded by the replay result.
- `metadata_boundary.excludes`: sensitive or raw content classes excluded from
  the replay result.
- `non_guarantees`: explicit boundaries such as no external audit, no
  certification, no legal compliance, no tamper-proof infrastructure, and no
  external SLSA certification.
- `known_limitations`: replay-specific limitations.

## Verification Rules

Replay JSON is valid only when:

- the referenced release packet exists;
- the packet commit can be resolved locally;
- Atlas can create an isolated replay checkout;
- the checkout reaches the packet commit;
- QA passes, unless `--skip-qa` is used and recorded as `skipped`;
- v1 readiness passes in the replay checkout;
- release packet verification passes from the replay checkout;
- `metadata_only` is `true`;
- `raw_artifacts_embedded` is `false`;
- no command logs or raw runtime artifacts are embedded.

## Allowed Statuses

`overall`:

- `verified`

`checks.*.status`:

- `ok`
- `skipped` for QA only when `--skip-qa` is used

`replay_checkout.cleanup`:

- `removed`
- `kept`

Failures are reported by a nonzero command exit and are not represented as a
successful replay JSON object.

## Metadata Boundary

Allowed:

- packet path
- release commit
- replay branch
- replay checkout path
- check statuses
- cleanup status
- command names
- timestamps
- known limitations
- non-guarantees

Forbidden:

- secrets
- credentials
- private keys
- tokens
- session cookies
- raw target data
- raw customer data
- payment data
- raw invoices or contracts
- packet captures
- full request or response bodies
- raw runtime artifacts
- unredacted evidence bodies
- full QA logs

## Non-Goals

This schema does not represent:

- not external audit
- not SLSA certification
- not legal compliance
- not enterprise deployment approval
- not tamper-proof infrastructure
- immutable storage
- a replacement for manual reviewer judgment

## Relationship To Release Trust

Release replay consumes `atlas.release_trust.v1` packets and complements:

- `atlas release verify`
- `atlas release manifest-verify`
- `atlas production status --strict`
- `atlas reviewer package`

Replay demonstrates that a retained release packet can still be checked from
the packet's recorded commit. It does not broaden the local Atlas production
contract or create an external certification claim.
